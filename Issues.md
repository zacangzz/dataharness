# Collection of Outstanding Issues

## Packaged worker subprocess launches TUI and times out (RESOLVED 2026-05-11)
- Observed in latest packaged run `dist/chats/w_0001/chat_f6e9bc91d1c0/messages.jsonl`: user asked `calculate sum of amount in sales`; the generated step was trivial pandas/pathlib code, but Layer 2 reported `execution timed out`.
- Worker telemetry showed the subprocess command was `dist/dataharness -m worker.sandbox_bootstrap <sandbox_config>`, followed 60 seconds later by timeout telemetry. In the prior implementation, `src/cli.py` ignored unknown `-m` args via `parse_known_args()` and built the TUI, so the child process never ran the sandbox bootstrap.
- Fix: added private CLI dispatch for `-m worker.sandbox_bootstrap <config>` before normal app parsing/building, with unknown private module targets failing fast, and explicitly packaged `worker.sandbox_bootstrap` as a PyInstaller hidden import. Increased default `ResourceLimits.timeout_seconds` from 60 to 120 seconds per user request, but kept that separate from the root fix.
- Sandbox policy remains unchanged: submitted worker code still cannot import `os`; Layer 3 `plan_analysis` rejects disallowed imports before approval, and Layer 2 reads only declared workspace-relative inputs while writing under `artifacts/tmp`.

## TUI file-picker selection drops prompt focus and transcript text is not easily copyable (RESOLVED 2026-05-11)
- Observed in the prompt bar after using the `@` file picker: selecting a file inserts the mention but focus can leave `#user_input`, forcing the user to click the prompt before continuing to type.
- Root cause: `PromptBar` moves focus into the `FilePicker` and then attempted to restore focus with a widget-local `self.set_focus(self.editor)` call that is not the reliable Textual focus API in this context; the exception was swallowed, so the app could end with no focused prompt.
- Related UX issue: Textual supports app-level text selection/copy, but DataHarness did not expose an explicit app copy action or fallback for focused panes, so conversation/sidebar/status text felt uncopyable even when rendered as text widgets.
- Fix: added `PromptBar._restore_editor_focus()` using `self.app.set_focus(self.editor)` with `editor.focus()` fallback; `FilePicker.Selected` and `FilePicker.Dismissed` now restore prompt focus. Added `DataHarnessApp.action_copy_text()` bound to `ctrl+c`/`super+c`, copying Textual selected text first and focused widget `text_buffer()`/rendered text as fallback. Covered by TUI regression tests.
- Fix pass 2026-05-11: made transcript message blocks focusable, surfaced the copy binding in the footer, and added a last-assistant-reply fallback so `Ctrl+C`/`Cmd+C` remains useful even when no transcript block is focused. Added regression coverage in `tests/app/tui/test_copy_text.py`.
- Fix pass 2026-05-11: Textual's clipboard is app-local and OSC52 is terminal-dependent, so universal copy/paste required a Layer 4 native clipboard provider instead of a mac-only patch. Added `src/app/tui/clipboard.py` with best-effort macOS/Windows/Linux command providers, kept Textual local clipboard as fallback, and added `Ctrl+V`/`Cmd+V` paste into the prompt editor only. Sidebar/status selection remains intentionally out of scope. Covered by `tests/app/tui/test_clipboard.py` and updated copy/paste action tests.

- Follow-up: delete `.git.broken-2026-05-10/` once confident nothing else needed from it; consider adding a remote so this is recoverable next time.
