# 📊 Sub-Spec 3: The Manifest & Data Schema (The "State")

**Date:** 2026-04-22  
**Status:** Draft (Pending User Review)

## 1. Overview
This specification defines the technical implementation of the **Manifest** and the **Data Schema**. 

The Manifest is the single source of truth for the entire system. It is a persistent, structured registry that captures the entire lifecycle of a data analysis session. While the **Brain** provides the intelligence and the **Worker** provides the muscle, the **Manifest** provides the **memory** and the **audit trail**. It ensures that every task, every decision, and every piece of data generated is recorded with full lineage, making the system inherently reproducible and auditable.

## 2. Technical Stack
* **Primary Format:** A single, structured `manifest.json` file (or a directory-based approach for very large sessions) that serves as the central registry.
* **Artifact Storage:** Data artifacts (e.g., `.parquet`, `.csv`, `.png`) are stored in a dedicated `artifacts/` directory, with their metadata and file paths strictly managed within the Manifest.
* **Schema Validation:** JSON Schema or Pydantic will be used to ensure the integrity of the manifest structure during reads and writes.

## 3. The Manifest Schema

The manifest is structured into three primary top-level registries to ensure clear separation of concerns.

### 3.1 Session Metadata
This section captures the "context" of the entire analytical session.
* **`session_id`**: A unique UUID for the session.
* **`start_timestamp`**: When the session began.
* **`user_context`**: Information about the user/environment.
* **`model_config`**: The model name and parameters used (e.g., `gemma-4-b`).
* **`version`**: The version of the harness used.

### 3.2 The Task Registry
A chronological list of every atomic task attempted during the session. Each entry must be immutable once marked as `completed` or `failed`.
* **`task_id`**: A unique identifier (e.g., `TASK-001`).
* **`timestamp`**: When the task was initiated.
* **`intent`**: A brief natural language description of what the task attempted to do.
* **`status`**: `planned` | `running` | `completed` | `failed` | `revised`.
* **`input_handles`**: A list of IDs or paths to artifacts used as input for this task.
* **`output_handles`**: A list of IDs or paths to artifacts generated by this task.
* **`execution_metadata`**: 
    * `code_snippet`: The exact Python code executed.
    * `error_log`: If failed, the captured traceback/error message.
    * `observation`: A summary of the findings/result as perceived by the Brain.
* **`parent_task_id`**: If this was a "Re-plan" or "Correction" task, a pointer to the task that triggered it.

### 3.3 The Artifact Registry
A centralized registry of all physical files and data objects generated during the session.
* **`artifact_id`**: A unique identifier (e.g., `ARTIFACT-001`).
* **`file_path`**: The relative path to the file in the `artifacts/` directory.
* **`type`**: The data type (e.g., `dataframe`, `plot`, `text_report`, `model_weights`).
* **`schema_definition`**: For data-heavy artifacts (like DataFrames), a lightweight schema summary (column names, types).
* **`lineage`**: A pointer to the `task_id` that produced this artifact.

## 4. Data Lineage & Versioning

The goal of this section is to ensure that the history of every data point is reconstructible.

### 4.1 The Lineage Chain
The manifest maintains a directed acyclic graph (DAG) of data generation.
* **Traceability:** Every entry in the `Artifact Registry` points to its `task_id`. 
* **Dependency Mapping:** The `Task Registry` tracks which `artifact_ids` were used as inputs. 
* **Result:** This allows the system (or a user) to trace any specific artifact (e.g., a final summary table) back through every single transformation and task that produced it.

### 4.2 Handling Re-planning & Re-execution
When the Brain decides to re-plan (e.g., due to an error or a change in direction), the manifest must preserve the history of the "failed" or "superseded" paths.
* **Immutable History:** We do not overwrite previous task entries. If `TASK-001` is superseded by `TASK-002`, both exist in the manifest. `TASK-001` is marked as `revised` or `superseded`, and `TASK-002` contains a pointer to it.
* **Versioned Artifacts:** If a task updates an existing dataset, a new `artifact_id` is created. The old artifact remains in the registry, allowing for "time-travel" debugging where we can inspect exactly what the data looked like at any stage of the process.

## 5. The Interaction Protocol (Brain $\leftrightarrow$ Manifest)

This section defines how the two active components (the Brain and the Worker) interact with the persistent state (the Manifest).

### 5.1 The Brain as the "Master of Truth"
The Brain is the only entity with write-access to the core structure of the Manifest.
* **Orchestrated Writes:** The Brain manages the creation of new tasks and the updating of task statuses.
* **Atomic Updates:** To prevent corruption, the Brain performs atomic writes. It prepares a complete updated version of the JSON structure and replaces the file in a single operation (or uses a temporary file + rename strategy).

### 5.2 The Worker's Role: "The Reporter"
The Worker does not write directly to the central `manifest.json` to avoid race conditions and maintain the Brain's control.
* **Result Hand-off:** When a task is completed, the Worker returns the results (e.g., file paths, success/fail status, and stdout/stderr) to the **Brain**.
* **The Brain's Responsibility:** Once the Brain receives the response from the Worker, *the Brain* updates the Manifest with the new task status and artifact registry entries.

### 5.3 Concurrency & Integrity
* **Single-Writer Model:** Since there is only one "Brain" process and one "Worker" process in a standard session, we maintain a single-writer model. The Brain is the sole writer.
* **Validation-on-Load:** Every time the Brain reads the Manifest, it must perform a schema validation check to ensure the file is intact and follows the defined structure. If the file is corrupted, the Brain triggers a "Recovery Mode" (e.g., attempting to roll back to a backup or notifying the user).

## 6. Success Criteria for Sub-Spec 3
* **Integrity:** The manifest can be validated against its schema at any time without loss of data.
* **Auditability:** A complete history of the session—from the first task to the final artifact—is preserved and can be reconstructed.
* **Reproducibility:** Given a specific `manifest.json`, the state of the data and the sequence of operations can be perfectly understood.

***

## 7. Summary: The Design Philosophy
The **Manifest** is the anchor of the entire system. It transforms a series of transient, ephemeral tasks into a permanent, structured record of discovery. By treating the state as a first-class, auditable object, we ensure that the "Brain" has a reliable memory and the user has a reliable audit trail.
